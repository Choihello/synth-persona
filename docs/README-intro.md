# synth-persona

### 통계 기반 합성 패널로 시장검증의 첫 관문을 좁히는 도구

---

## 문제: 조사를 못 하는 게 아니라, 조사를 시작할 준비가 안 된 것

초기 창업자에게 "사용자 검증을 하라"는 조언은 누구나 한다. 문제는 방법론이 아니라 진입 장벽이다. 패널 모집에는 보통 2주 이상이 걸리고, 제대로 된 정성조사 한 번에 수백만 원이 들어간다. 그 비용을 감당할 수 없는 초기 단계 창업자는 결국 "감"으로 의사결정을 하게 되고, 그 감이 틀렸다는 걸 알게 되는 시점은 이미 제품을 만들고 난 뒤다.

더 근본적인 문제는 따로 있다. 막상 예산이 생겨 조사를 하려 해도 "누구에게, 무엇을 물어야 하는지"가 명확하지 않은 경우가 대부분이다. 타깃이 모호한 채로 설문지부터 만들면, 응답률은 낮고 응답의 해석 가능성은 더 낮다. 즉 진짜 병목은 조사 실행 능력이 아니라 **조사를 설계할 만큼 가설이 좁혀져 있지 않다는 것**이다.

## 기존 대안들의 한계

이 문제에 대한 기존 해법은 크게 두 갈래다. 하나는 전통적 시장조사 — 정확하지만 느리고 비싸서 초기 단계에 맞지 않는다. 다른 하나는 최근 등장한 "AI가 소비자 반응을 예측해준다"류의 도구들인데, 여기엔 더 근본적인 문제가 있다. 이런 도구 대부분은 LLM에게 막연히 "30대 직장인처럼 답해줘"라고 요청한 뒤 그 결과를 시장 예측처럼 제시한다. 이때 두 가지가 숨겨진다 — 첫째, 그 "30대 직장인"이라는 페르소나가 실제 인구 분포를 반영하는지 아무도 검증하지 않는다. 둘째, LLM 응답 자체의 변동성(예스맨 편향, 평균 회귀, 프롬프트 민감도)이 그대로 결과에 섞여 들어가는데도 이를 구분해서 보여주지 않는다. 결과적으로 "정확해 보이지만 실제로는 검증 불가능한 숫자"가 양산된다.

synth-persona는 이 업계의 흔한 함정 — **그럴듯함과 검증 가능성을 혼동하는 것** — 을 피하는 데서 출발한다.

## synth-persona가 하는 일

### 1층 — 합성 인구 구성

통계청 2024 인구총조사 데이터(약 4,346만 명)를 IPF(반복비례조정)로 결합해, 성×연령×권역 3축에 대해서는 실측 교차표를 그대로 가중 열거(matched-core)하고, 혼인상태·가구원수는 연령을 앵커로 한 조건부 분포를 부착한다. "완전한 5속성 결합분포"라고 주장하지 않는다 — matched-core 3축과 conditioned 2축을 명시적으로 구분하고, 가구원수의 경우 가구주 연령 표를 개인 단위로 끌어오는 근사(`householder_age_as_proxy`)를 썼다는 사실까지 페르소나 메타데이터에 남긴다. 통계청 비공표값(`X`, 구조적 결측이 아닌 비공개)과 해당없음(`-`, 구조적 0)도 구분해서 처리한다 — 이 구분을 흐리면 합성 인구 자체가 왜곡된다.

### 2층 — LLM 응답 시뮬레이션

합성 인구에서 가중치 비례로 표본을 추출해 LLM(기본값 Claude Haiku, 비용 효율 우선)에 질문을 던진다. 각 페르소나의 system 프롬프트에는 속성값뿐 아니라 그 속성의 출처(matched/conditioned)와 근사 여부가 함께 들어간다 — 모델이 "확실한 것"과 "추정한 것"을 구분해서 응답하도록.

### 집계 — 합의와 분열을 분리해서 본다

전체 응답을 정규화 엔트로피로 집계해 🔴(분열)/🟢(합의) 신호로 표시하고, 모든 인구 속성 축에 대해 세그먼트 교차표를 자동 생성한다. 핵심은 "전체는 갈려도 특정 세그먼트 안에서는 합의가 보인다"는 패턴을 잡아내는 것이다 — 이게 바로 "조사를 어디에 조준해야 하는지"의 신호다.

## 이 도구를 쓰면 실제로 무엇을 얻는가

질문 하나를 던지면 다음 네 가지를 구조화된 형태로 받는다.

| 받는 것 | 무엇을 알려주는가 |
|---|---|
| **전체 신호** | 응답이 합의(🟢)됐는지 분열(🔴)됐는지, 분열의 정도(분산) |
| **세그먼트 교차표** | 성별·연령·지역·혼인상태·가구원수 등 모든 축에서, 그 세그먼트 *안*에서는 의견이 갈리는지 합의되는지 |
| **신뢰성 카드** | 통계 구성이 얼마나 정확한지(1층) · 어느 속성이 실측이고 어느 속성이 추정인지(2층) · 이 결과로 어떤 판단을 내리면 **안 되는지**(4층) |
| **누락 응답 기록** | 응답 실패·매칭 실패 건수와 비율 — 조용히 숫자가 왜곡되는 것을 막기 위함 |

즉 "정답"을 받는 게 아니라, **이 가설을 검증하려면 어디를 더 봐야 하는지에 대한 구조화된 진단**을 받는다. "전체는 의견이 갈렸지만 20·30대 안에서는 합의가 보인다" 같은 식으로 — 다음 인터뷰에서 누구를 만나야 할지가 바로 도출된다.

## 신뢰성을 층으로 분리한다

| 층 | 무엇을 보여주는가 | 측정 방식 |
|---|---|---|
| **구성 신뢰도** | 합성 인구가 통계청 원본 분포를 얼마나 정확히 재현하는가 | MAE/TVD, 자기검증 가능 |
| **속성 신뢰도** | 이 결과가 실측(matched) 속성에서 나왔는가, 추정(conditioned/inferred) 속성에서 나왔는가 | provenance 추적, 혼합 시 가장 보수적인 값 채택 |
| **응답 신뢰도** | LLM 응답 자체가 얼마나 일관적인가 | 자기일관성·예스맨 편향·평균회귀 탐침(프레임 구축됨, 실측 배선은 다음 단계) |
| **시장판단 신뢰도** | 이 결과를 **어디에 쓰면 안 되는가** | 점수가 아니라 부정형 가드레일 |

네 번째 층이 가장 중요하다. 결과가 그럴듯해 보일수록 "이건 믿어도 되는 결과처럼 보인다"는 신호를 의도적으로 차단해야 하기 때문이다 — 예를 들어 소득·직업·자녀 유무 축이 없는 현재 합성 인구로는 가격이나 구매력 판단을 내릴 수 없다는 사실을 매 결과마다 명시한다.

## 무엇을 하지 않는가

- 정량적 시장 예측을 제시하지 않는다. 모든 응답 수치는 시장 예측이 아니라 **synthetic panel response**(가상 패널 응답)로 라벨링되며, "구매율 34%"가 아니라 "가상 패널 응답: 34/100, 실제 구매율 아님" 형태로만 표기한다.
- 조사를 대체한다고 주장하지 않는다. 의견이 갈리는 지점을 짚어 실제 조사 예산을 어디에 써야 하는지 가리킬 뿐이다.
- 추정값을 실측값과 같은 무게로 다루지 않는다. 모든 속성에 출처(provenance)를 표기하고, 약한 추정에서 나온 결론은 신뢰도를 낮춰 보여준다.

## 사용 예시

세 가지 진입점이 있다. 아래 예시는 모두 **결정적 mock 응답**으로 돌아간다 — 합성 인구 구성(1층)과 파이프라인 구조(세그먼트 분해·신뢰성 카드)가 정직하게 동작함을 키 없이 그대로 확인할 수 있다는 뜻이다. 다만 이건 **실제 LLM 응답(2층)이 아니다.** 가상 패널이 진짜로 어떻게 답하는지 보려면 `MockProvider`를 `ClaudeProvider`로 바꾸면 되고, 이때부터는 `ANTHROPIC_API_KEY`가 필요하다.

### 1) CLI로 빠르게 — 통계청 합성 인구에 질문 던지기

```bash
node dist/cli/main.js --question "신선식품 새벽배송 구독, 월 9900원에 쓸 의향?" \
    --choices "쓴다,안쓴다" --n 60 --seed 7 --source census --mock
```

```
전체 신호: 🔴 split (분산 0.90)
응답 분포: 안쓴다=41, 쓴다=19

[성별]
  🔴 남자: 안쓴다=24, 쓴다=10
  🔴 여자: 안쓴다=17, 쓴다=9

[연령별]
  🟢 15~19세: 안쓴다=2
  🟢 25~29세: 쓴다=5
  ... (15개 연령 구간 · 혼인 · 가구원수 세그먼트 계속)
```

전체는 분열돼도(🔴) 특정 연령·가구원수 세그먼트 안에서는 합의(🟢)가 보인다 — 어디를 조준해 진짜 인터뷰를 해야 할지가 한눈에 드러난다.

### 2) 신뢰성 카드까지 — 결과를 얼마나 믿어도 되는지 확인

```bash
npm run reliability:demo
```

```markdown

### 1층 · 구성 신뢰도 (통계청 분포 적합)
- 🟢 MAE 0.0000 · TVD 0.0000

### 2층 · 속성 신뢰도 (provenance)
| 속성 | provenance | 신뢰도 | 비고 |
|---|---|---|---|
| 성 | matched | 높음 |  |
| 연령 | matched | 높음 |  |
| 지역 | matched | 높음 |  |
| 혼인 | inferred | 낮음 |  |
| 가구원수 | conditioned | 중간 | bridge:householder_age_as_proxy |

### ⚠️ 가드레일 — 이 결과로 하지 말 것
- 이 숫자는 synthetic panel response입니다 — 실제 구매율/시장 예측이 아니며, 실제 인터뷰·설문으로 검증해야 합니다.
- 가격·구매력 판단 부적합: 소득·직업·자녀 축이 없습니다. 방향 가설 탐색엔 사용 가능하나, 가격 결론은 인터뷰로 확인하세요.
- 저확신 속성(혼인)에서 나온 세그먼트 결론은 추정(inferred)이므로 의사결정에 쓰지 마세요.
```

`혼인` 속성이 `inferred`(낮음)로 강등된 건 표본에 혼인 통계 대상 연령(18세 미만) 밖 페르소나가 섞여 있기 때문이다 — 이런 식으로 신뢰도가 자동으로, 그리고 보수적으로 매겨진다.

### 3) 라이브러리로 직접 — 코드에서 호출

```ts
import {
  CensusPopulation,
  runCensusStudy,
  MockProvider, // 키 없는 데모용
} from "synth-persona";
import snapshot from "synth-persona/data/census/kr-2024.json" with { type: "json" };

const population = new CensusPopulation(snapshot);

const result = await runCensusStudy({
  population,
  provider: new MockProvider((p) => (p.attrs.연령 < "40" ? "끌린다" : "안 끌린다")),
  // 실제 응답을 보려면: provider: new ClaudeProvider() — ANTHROPIC_API_KEY 필요
  question: { prompt: "이 컨셉 어때요?", choices: ["끌린다", "안 끌린다"] },
  n: 100,
  seed: 42,
});

console.log(result.signal);     // "consensus" | "split"
console.log(result.bySegment);  // 세그먼트별 신호 + 분포
```

## 검증 가능성

synth-persona는 스스로를 검증하는 하네스를 함께 제공한다 — IPF 불변식을 무작위 분포에 대해 property-test로 검증하고, 같은 시드는 항상 같은 결과를 내는 결정성을 테스트하며, 과거 사례에 대한 캘리브레이션 백테스트로 순위상관·오차·방향정확도를 성적표로 낸다. 합성 인구 자체도 통계청 원본과의 재집계 오차(현재 MAE 0에 가까움)로 검증된다. **이 모든 검증은 API 키 없이 mock/VCR 기반으로 재현 가능하다** — "이 도구가 실제로 주장대로 동작하는지"를 누구나 직접 확인할 수 있다는 뜻이고, 이게 다른 "AI 시장조사" 도구들과 가장 분명하게 갈리는 지점이다.

```bash
npm test               # 138개 테스트, 키 불필요
npm run fidelity:demo  # 1층 합성인구 충실도
npm run calibrate:demo # 캘리브레이션 성적표
```

## 누구를 위한 도구인가

쓸모 있는 범위는 명확하다 — 정량 예측이나 가격 탄력성 같은 정밀한 숫자가 필요한 의사결정이 아니라, **탐색적 정성조사, 컨셉 간 상대 비교, 세그먼트별 태도의 결을 잡고, 진짜 설문을 설계하기 전 가설을 한 번 좁히는 0차 단계**다. 그 이상을 약속하지 않는 것이 이 도구의 설계 원칙이다.

synth-persona는 오픈소스(MIT)로 공개되어 있으며, 통계청 2024 인구총조사 스냅샷이 번들로 포함돼 있어 API 키 없이도 전체 파이프라인을 그대로 재현해볼 수 있다.
