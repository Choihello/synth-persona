# synth-persona 제품 재정렬 설계 (product reorientation) — 2026-06-30

> 핵심 취지를 다시 정렬한다. 이 도구는 **"AI가 시장조사를 대체한다"가 아니라, 실제 조사를 하기 전에 가설을 압축하는 사전 시장검증 준비 도구**다. 본 문서는 그 재정렬을 north-star로 정식화하고, 구현을 **묶음 A(키 불필요, 이번 사이클)** 와 **묶음 B(2층·키 필요, 나중)** 로 나눈다. 구현 플랜(writing-plans)은 **묶음 A만** 다룬다.

---

## 0. 한 줄 정의

초기 창업자가 감으로 넘기던 **시장검증 전 단계**를 〈통계 기반 가상 패널 + LLM 반응 + 신뢰도 카드 + 실제 검증 액션〉으로 구조화하는 도구. **정답 생성기가 아니라, 첫 고객 대화와 첫 검증 실험을 설계해주는 도구.**

## 1. 포지셔닝 (불변 원칙)

이미 README에 있는 명제를 **제품 원칙으로 승격**해 박제한다. 앞으로 모든 구현·문구는 이 원칙에 자기일관적이어야 한다.

- **실제 조사의 대체재가 아니라, 실제 조사 전 가설을 압축하는 도구.**
- **정답을 주는 도구가 아니라, 무엇을 누구에게 검증해야 하는지 알려주는 도구.**
- **막연한 아이디어를 → 세그먼트 · 반응 가설 · 거부 이유 · 인터뷰 질문 · 검증 액션으로 바꿔주는 도구.**

### 1.1 문구 가드 (제품 카피)

| ❌ 나쁜 문구 (단정·예측) | ✅ 좋은 문구 (가설·조준) |
|---|---|
| "설문 없이 시장조사 완료" | "실제 조사 전에 시장 가설을 빠르게 압축합니다" |
| "소비자 반응 예측" | "누구를 만나고 무엇을 물어볼지 제안합니다" |
| "AI가 구매율을 알려줍니다" | "통계 기반 가상 패널로 초기 반응과 위험 신호를 탐색합니다" |
| "한국 소비자 42%가 삽니다" | "이 세그먼트에서 관심 가능성이 높고, 이 부분은 인터뷰로 검증해야 합니다" |

**규칙**: 출력의 모든 숫자는 *시장 예측*이 아니라 **synthetic panel response**(가상 패널 응답)로 라벨링한다. "구매의향 34%"는 금지, "가상 패널 응답: 34/100 (실제 구매율 아님)"으로 표기.

## 2. 4층 신뢰성 모델

신뢰성을 **층으로 분리**해서 보여준다. 측정 가능한 층과 무른 층을 섞지 않는다.

| 층 | 무엇 | 측정 가능? | 묶음 | 소스 |
|---|---|---|---|---|
| **1. 구성 신뢰도** | 합성 인구 ↔ 통계청 분포 적합도 | ✅ MAE/TVD | **A** | 기존 `populationFidelity` |
| **2. 속성 신뢰도** | 속성이 matched / conditioned / inferred 중 무엇인가 | ✅ provenance | **A** | 기존 `persona.provenance` |
| **3. 응답 신뢰도** | LLM 응답이 얼마나 일관적인가 | ✅ probes | A는 *훅만*, 실측은 **B** | 기존 `probes`(배선은 B) |
| **4. 시장판단 신뢰도** | 이 결과를 어느 의사결정에 써도 / **쓰지 말아야** 하나 | ❌ **부정형 가드레일** | **A** | 파생(축 커버리지 규칙) |

### 2.1 핵심 결정

- **1·2·4층은 키 없이 묶음 A에서 노출.** 3층은 실제 LLM 호출이 필요한 실측이므로 묶음 A에서는 *자리(placeholder)만* 잡고, 묶음 B에서 probes를 배선한다.
- **4층은 점수가 아니라 부정형 가드레일이다.** "이 결과 = 의사결정 가능" 같은 green light는 그 자체가 과장(원칙 1·5 위반)이 된다. 따라서 4층은 **"무엇에 쓰면 안 되는가"** 만 말한다.
  - 예: *"가격·구매력 판단 부적합 — 소득·직업·자녀 축이 없음. 방향 가설 탐색엔 사용 가능. 반드시 실제 인터뷰로 확인."*
  - 축 커버리지에서 **정적 파생**한다. 질문 의미 분석(질문이 가격에 관한 것인지 인지)은 LLM이 필요하므로 **묶음 B로 이월**.

### 2.2 1층/2층 비혼합 (불변식)

통계청 기반으로 방어 가능한 것은 **"이런 속성 조합의 가상 패널을 구성했다"는 1층**뿐이다. LLM이 답한 내용은 **2층 추론**이며, 항상 불확실성/주의 신호를 같이 보여준다. 출력에서 두 층을 **시각적으로 분리** 표기한다.

## 3. 출력 비전: 진단 → 처방

현재 도구의 출력은 **진단**(전체/세그먼트 🔴 split / 🟢 consensus 교차표 + "여기 조사 권장" 한 줄)에서 멈춘다. 제품의 알맹이는 **처방**(다음 행동)이다.

- **좋은 출력 = 다음 행동**: 가장 반응할 세그먼트 / 끌릴 이유 / 거부 이유 / 병목(가격·신뢰·습관·대체재) / 만날 고객 유형 / 인터뷰 질문지 / 설문 문항 초안 / 랜딩 메시지 / 가장 불확실한 가정.
- 진단은 묶음 A에서 **신뢰도 카드**로 정직하게 보강하고, 처방은 묶음 B(2층)에서 생성한다.

## 4. 묶음 A — 신뢰성 표현 강화 (키 불필요, 이번 사이클)

### 4.1 새 모듈 `src/assess/reliability.ts` (순수 함수, read-only 오버레이)

코어 타입·`aggregate`를 **건드리지 않는다**(기존 89 테스트 불변). `StudyResult`를 **읽기만** 한다 — provenance는 이미 `response.persona.provenance`에 있어, 세그먼트 dim의 provenance를 역추적할 수 있다.

```ts
assessReliability(result: StudyResult, ctx: {
  snapshotMeta?: SnapshotMeta;     // year, sources (provenance 출처)
  fidelity?: FidelityReport;       // 1층 (있으면)
  axes: string[];                  // 모집단이 실제 가진 축: 성·연령·지역·혼인·가구원수
}): ReliabilityCard
```

```ts
interface ReliabilityCard {
  composition: { signal: "🟢" | "🔴"; mae: number; tvd: number } | null; // 1층
  attributes: {                                                          // 2층 (세그먼트 dim별)
    dim: string;
    provenance: "matched" | "conditioned" | "inferred";
    confidence: "high" | "medium" | "low";
    note?: string;                 // 예: bridge:householder_age_as_proxy
  }[];
  responseConsistency: { status: "not-measured"; reason: "키 필요(묶음 B)" }; // 3층 자리
  guardrails: string[];            // 4층 부정형 — 예: "가격/구매력 판단 부적합: 소득·직업·자녀 축 없음"
  missingAxes: string[];           // 알려진 결핍 축: ["소득","직업","자녀"] 등
}
```

### 4.2 규칙 (전부 정적·키 불필요)

1. **provenance → confidence 매핑**: `matched`=high, `conditioned`=medium(+bridge면 note 첨부), `inferred`=low. → 세그먼트별로 "이 갈림이 얼마나 믿을 만한가"가 붙는다. (예: `연령`에서 갈린 🔴 > `가구원수`에서 갈린 🔴.)
2. **축 결핍 가드레일**: 모집단 축에 소득/직업/자녀가 없으면 → "가격·구매력 결론 저신뢰" 가드레일을 **항상** 표기. (현재 축: 성·연령·지역·혼인·가구원수.)
3. **숫자 라벨링**: 렌더러가 응답 카운트를 **"synthetic panel response (실제 구매율 아님)"** 로 라벨.
4. **1층 노출**: `fidelity`가 주어지면 구성 신뢰도 한 줄(🟢/🔴 + MAE/TVD).

### 4.3 렌더러

`renderReliabilityCard(card): string`(markdown). 기존 `renderMarkdownReport`에 **선택적 reliability 섹션**으로 결합한다(인자 옵셔널 → 기존 호출 불변). 1층(구성)과 2층(LLM 응답)을 **명확히 분리된 블록**으로 렌더한다.

> **CLI 배선은 묶음 A 범위 밖이다(묶음 B로 이월).** 현재 CLI는 provenance 없는 `SampleSource` 경로라 신뢰성 카드가 대부분 `unknown`이 되어 가치가 낮다. 묶음 A는 `reliability:demo`(키 없는 census 기반)로 신뢰성 카드를 보여주고, CLI 배선은 `runStudy`에 `PersonaSource`/census 경로가 연결되는 묶음 B에서 처리한다. (plan의 "범위 밖" 설명과 동일.)

### 4.4 공개 API

`src/index.ts` 배럴에 `assessReliability`, `renderReliabilityCard`, 타입 `ReliabilityCard` 추가.

## 5. 묶음 B — 진단에서 처방으로 (2층·키 필요, 나중)

> 본 사이클에서 **구현하지 않는다.** 비전으로만 기록. 별도 brainstorming/spec 사이클 + `ANTHROPIC_API_KEY` 선결.

- **다음 행동 생성물 (2층 LLM)**: 끌릴 이유 / 거부 이유 / 병목 분류(가격·신뢰·습관·대체재) / 만날 고객 유형 / 인터뷰 질문지 / 설문 문항 초안 / 랜딩 메시지 / 가장 불확실한 가정.
- **3층 응답 신뢰도 실측 배선**: probes(자기일관성·예스맨·평균회귀)·robustness를 study 경로에 연결.
- **질문 종류 인지 가드레일**: 질문이 "가격"에 관한 것이면 소득축 결핍을 더 강하게 경고(LLM 분류).

## 6. MVP 성공 기준

완벽한 수요 예측기가 아니어도 된다. 창업자가 결과를 보고 다음을 얻으면 성공:

- 1차 타깃이 누구인지 더 선명해진다.
- 예상 반대 이유를 알게 된다.
- 실제 인터뷰/설문 질문을 바로 만들 수 있다. *(묶음 B)*
- 랜딩 메시지·가격 가설을 테스트할 방향이 생긴다. *(묶음 B)*
- 어떤 판단은 아직 믿으면 안 되는지 알게 된다. *(묶음 A — 신뢰도 카드·가드레일)*

## 7. 테스트 / 불변식 (묶음 A)

- **새 모듈 TDD**: provenance→confidence 매핑, 축 결핍 가드레일, `inferred`=low, bridge note, 빈 결과 방어.
- **렌더러 테스트**: 숫자 라벨("synthetic panel response") 포함 확인, 1층/2층 분리 블록 확인.
- **기존 89 테스트 불변** + lint/tsc/build 그린.
- **설계 불변식 준수**:
  - 1층(구성)과 2층(LLM 응답)을 **시각적으로 분리** 표기.
  - 모든 숫자 = **synthetic panel response** 라벨(실제 구매율 아님).
  - **conditioned/inferred 결론 = 저확신** 표시.
  - **"완전 5-way joint" 비주장** (matched-core 3축 + 조건부 2).
  - `Persona.weight` · provenance(matched/conditioned/inferred) · householder bridge · suppressed(X)≠structural zero(-) 유지.

## 8. 범위 밖 (YAGNI)

- 질문 의미 분석/분류 (묶음 B).
- 소득·직업·자녀 등 신규 축 추가 (별도 데이터 사이클 — 본 스펙은 *결핍을 정직히 표기*만).
- 시장판단 신뢰도의 수치 점수화 (의도적으로 부정형 가드레일만).
